# ФОЛИО WinСклад - бизнес-процесс "Счет" и интеграция WooCommerce

## Статус документа

Версия: 0.2

Назначение: зафиксировать бизнес-логику работы со счетами ФОЛИО и безопасный план интеграции WooCommerce с уже подготовленным endpoint счета.

Важно: этот документ не является финальным контрактом JSON. Все поля, которые отправляются в ФОЛИО, должны быть отдельно согласованы. Нельзя заполнять справочные значения наугад.

## Цель интеграции

Связать документы WooCommerce с документами ФОЛИО так, чтобы:

- для каждого созданного в ФОЛИО счета сохранялся уникальный номер/ID документа;
- этот номер был привязан к соответствующему заказу или черновику Woo;
- черновики Woo создавали в ФОЛИО неучитываемый счет;
- реальные заказы Woo создавали в ФОЛИО учитываемые счета по нужным складам;
- Woo не пытался самостоятельно угадывать справочные значения ФОЛИО;
- перед включением отправки была возможность увидеть preview JSON.

## Подтвержденная модель ФОЛИО

### Основные таблицы

| Таблица | Назначение | Статус |
|---|---|---|
| `SCL_NAKL` | шапка документа | подтверждено |
| `SCL_MOVE` | строки документа | подтверждено |
| `SCL_ARTC` | товар/остатки по складам | подтверждено |
| `SCL_ADDN` | дополнительные реквизиты счета | частично подтверждено |
| `SCL_PLAT` | платежи | исследовать позже |
| `SCL_PMOV` | строки платежей | исследовать позже |

Связь документа:

```text
SCL_NAKL.UNICUM_NUM
        |
        v
SCL_MOVE.UNICUM_NUM
```

### Остатки

В `SCL_ARTC` используются два важных поля:

- `KON_KOLCH` - фактический остаток;
- `REZ_KOLCH` - свободный/доступный остаток.

Подтвержденное поведение учитываемого счета:

- создание счета не меняет `KON_KOLCH`;
- создание счета уменьшает `REZ_KOLCH`;
- изменение количества меняет резерв на дельту;
- удаление строки возвращает резерв;
- несколько счетов суммируют резерв.

Неучитываемый счет должен быть аналогом черновика/перспективного заказа и не должен влиять на складские остатки.

## Текущий API ФОЛИО

Базовый путь:

```text
/admin/folio/accounts
```

API работает с документом типа `СЧЕТ`.

Ключевые операции:

| Операция | Endpoint | Назначение |
|---|---|---|
| Создать счет | `POST /admin/folio/accounts` | создать документ и строки |
| Получить счет | `GET /admin/folio/accounts/{unicumNum}` | прочитать шапку, доп. реквизиты и строки |
| Список счетов | `GET /admin/folio/accounts?...` | поиск счетов |
| Изменить количество строки | `PATCH /admin/folio/accounts/{unicumNum}/items/{recno}` | изменить резерв по строке |
| Добавить строку | `POST /admin/folio/accounts/{unicumNum}/items` | добавить товарную строку |
| Удалить строку | `DELETE /admin/folio/accounts/{unicumNum}/items/{recno}` | вернуть резерв и удалить строку |
| Удалить счет | `DELETE /admin/folio/accounts/{unicumNum}` | сейчас отключено, `501` |
| Мягкая отмена | `POST /admin/folio/accounts/{unicumNum}/cancel` | служебный путь, не считать финальным API удаления |

Источник подробного API-контракта:

```text
docs/api/FOLIO_ACCOUNT_JS_API.md
```

### Важные правила API

- `externalRequestId` обязателен и должен быть стабильным при повторе одного и того же запроса.
- `documentNumber` сейчас должен быть числовым, потому что `SCL_NAKL.N_PLAT_POR` имеет тип `float NOT NULL`.
- Для создания использовать только `operationType: "СЧЕТ"`.
- `documentDate` можно передавать как `LocalDateTime`, но ФОЛИО нормализует дату до `00:00:00`.
- `TYPE_DOC` и `SCL_MOVE.TYPDOCM_PR` для счета пишутся как кириллическая `С`, не латинская `C`; JS/Woo это не должны передавать.
- `VID_DOC` - это вид операции из справочника ФОЛИО, а не технический тип документа.
- После `409 INSUFFICIENT_AVAILABLE_STOCK` нельзя повторять тот же запрос без изменения количества/остатков.

## Справочные поля ФОЛИО

Часть данных нельзя генерировать в Woo автоматически. Они должны быть выбраны из справочников ФОЛИО или явно привязаны в админке Woo.

| Поле API | Справочник/таблица ФОЛИО | Правило |
|---|---|---|
| `warehouseId` | `SCLAD_R.ID_SCLAD` | склад должен существовать |
| `sku` | `SCL_ARTC.COD_ARTIC` + `SCL_ARTC.ID_SCLAD` | товар должен существовать на выбранном складе |
| `folioOperationKind` | `VID_OPER`, поле `VID_DOC` | выбирать из справочника, не фиксировать константой |
| `priceContractType` | `_KONTRCT`, `CONTR_POR`/`CONTRACT_N` | выбирать из справочника цены/договора |
| `payerName` | `_PARTNER`, полное имя | организация/плательщик |
| `payerShortName` | `_PARTNER`, краткое имя | важный идентификатор клиента ФОЛИО |
| `receiverName` | справочник моих организаций | получатель/моя организация |
| `sourceInfo` | `_RECLAMA` | источник информации |
| `additionalInfo` | `DOP_INF` | дополнительная информация |
| `folioUser` | пользователи ФОЛИО | автор/редактор документа, сейчас рабочий пользователь `buh` |

## Привязки, которые нужны в Woo

### Woo client -> Folio client

Нужна админская настройка соответствия клиента Woo клиенту ФОЛИО.

Минимально хранить:

- Woo user ID или другой ключ клиента Woo;
- Folio `payerShortName`;
- Folio `payerName`;
- при необходимости `payerCity`, `payerPhone`, `directorName`, `accountantName`, `deliveryInfo`;
- выбранный `priceContractType`;
- выбранный `sourceInfo`;
- выбранный `additionalInfo`;
- выбранный `receiverName`.

Принцип: не угадывать клиента по строке имени при создании счета. Если привязки нет, показывать ошибку/предупреждение и не отправлять счет в ФОЛИО.

### Woo location -> Folio warehouse + priority

Сейчас Woo использует таксономию `location` и план списания `_pc_alloc_plan`.

Для реальных счетов нужна настройка:

- Woo location term ID;
- Folio `warehouseId`;
- название склада ФОЛИО для UI;
- приоритет склада внутри группы Киев/Одесса;
- признак активности.

Назначение: если Woo план говорит "Киев" или "Одесса", интеграция должна понимать, на какой склад ФОЛИО создавать строку/счет и как распределять приоритеты, если внутри города несколько складов.

## Что хранить в Woo order meta

Meta заказа не должна быть копией полного JSON счета. В Woo нужно хранить связь и диагностическое состояние.

Рекомендуемые meta:

| Meta key | Назначение |
|---|---|
| `_folio_doc_id` | `SCL_NAKL.UNICUM_NUM` или другой главный ID документа |
| `_folio_doc_number` | номер счета ФОЛИО |
| `_folio_doc_type` | например `draft_account` или `account` |
| `_folio_accounting_enabled` | учитываемый/неучитываемый документ |
| `_folio_sync_status` | `not_synced`, `previewed`, `synced`, `failed`, `partial` |
| `_folio_sync_error` | последняя ошибка отправки |
| `_folio_synced_at` | дата последней успешной синхронизации |
| `_folio_payload_hash` | хэш последнего согласованного payload |
| `_folio_external_request_id` | стабильный idempotency key запроса |
| `_folio_customer_ref` | выбранная привязка клиента ФОЛИО |

Полный payload можно хранить только как debug/log-снимок, но не использовать его как основной источник истины.

## Текущие Woo-потоки

### Наполнение корзины

Корзина может наполняться из нескольких мест:

- карточка товара и каталог через обычный Woo add-to-cart;
- AJAX добавление на странице товара;
- каталожные qty-кнопки;
- быстрый оптовый заказ;
- импорт файла в корзину.

Эти входы не должны напрямую создавать документы ФОЛИО. Они только формируют корзину Woo.

### Импорт файла в корзину

Код:

```text
wp-content/plugins/pc-order-import-export/inc/ImporterCart.php
```

AJAX:

```text
pcoe_import_cart
```

Поведение:

- читает CSV/XLS/XLSX;
- определяет товар по GTIN или SKU;
- добавляет товар в корзину;
- цены из файла не учитывает.

ФОЛИО: ничего не отправлять.

### Корзина -> черновик Woo

Код:

```text
wp-content/plugins/pc-order-import-export/inc/CartToDraft.php
```

AJAX:

```text
pcoe_cart_to_draft
```

Поведение:

- создает Woo order со статусом `wc-pc-draft`;
- переносит товары из корзины;
- штампует `_pc_alloc_plan`;
- сохраняет order;
- очищает корзину.

ФОЛИО:

- создать неучитываемый счет;
- не разбивать заказ по складам;
- сохранить ID/номер счета ФОЛИО в meta Woo order;
- если обязательных привязок клиента нет, не отправлять и показать понятную ошибку.

### Импорт файла -> черновик Woo

Код:

```text
wp-content/plugins/pc-order-import-export/inc/ImporterDraft.php
```

AJAX:

```text
pcoe_import_order_draft
```

Поведение:

- создает Woo order со статусом `wc-pc-draft`;
- добавляет строки из файла;
- может принимать цену из файла;
- возвращает отчет импорта.

ФОЛИО:

- тот же сценарий, что "Корзина -> черновик Woo";
- создать неучитываемый счет;
- сохранить связь с ФОЛИО в meta Woo order.

Технический риск: текущий вызов штамповки плана в `ImporterDraft.php` выглядит ошибочным и должен быть проверен отдельным small patch перед интеграцией.

### Черновик/заказ -> корзина

Код:

```text
wp-content/plugins/pc-order-import-export/inc/DraftToCart.php
```

Поведение:

- добавляет строки заказа обратно в корзину;
- может очистить текущую корзину перед добавлением.

ФОЛИО:

- на первом этапе ничего не делать;
- позже можно добавить read-only проверку: есть ли связанный документ ФОЛИО и совпадает ли он с Woo.

### Checkout -> реальный Woo order

Код:

```text
wp-content/plugins/paint-core/inc/header-allocation-switcher.php
wp-content/plugins/paint-core/inc/order-allocator.php
```

Поведение:

- при checkout переносит или рассчитывает `_pc_alloc_plan` для строк заказа;
- текущий план списания зависит от режима allocation: auto/manual/single;
- Woo стандартное списание отключено;
- локальное custom-списание сейчас выполняется на статусах `processing` и `completed`.

ФОЛИО:

- это главный поток для создания реальных учитываемых счетов;
- счета должны создаваться по нужным складам ФОЛИО;
- если один Woo order распределен по нескольким складам ФОЛИО, возможны несколько счетов ФОЛИО;
- нужно сохранить связь Woo order -> один или несколько Folio document IDs;
- если часть товара не найдена/не может быть отправлена, заказ становится частичным: это отдельный TODO-сценарий.

## Бизнес-правила для черновика ФОЛИО

Черновик Woo (`wc-pc-draft`) должен соответствовать неучитываемому счету ФОЛИО.

Правила:

- не влияет на остатки ФОЛИО;
- может не разбиваться по складам;
- нужен для перспективного заказа/предзаказа;
- должен иметь стабильный `externalRequestId`;
- в Woo должен сохраняться номер/ID созданного документа ФОЛИО;
- повторная отправка того же черновика не должна создавать дубль.

Открытый вопрос: какой точный флаг API отвечает за неучитываемость. В API-документе есть `accountingEnabled`; предварительно для черновика ожидается `false`, но это нужно подтвердить экспериментом.

## Бизнес-правила для реального счета ФОЛИО

Реальный счет создается при переходе Woo order на выполнение.

Предварительная логика:

1. Woo order создан из checkout.
2. Для строк заказа есть `_pc_alloc_plan`.
3. Интеграция группирует строки по Folio warehouse.
4. Для каждой группы создает учитываемый счет ФОЛИО.
5. Возвращенные Folio document IDs сохраняются в Woo order meta.
6. Если все счета созданы успешно, Woo order можно переводить дальше по процессу.
7. Если часть строк не отправилась, помечать заказ как частичный и показывать предупреждение.

## Формирование JSON

JSON должен формироваться на сервере Woo/PHP, а не в браузерном JS.

Причины:

- JS не должен знать служебные справочники и правила ФОЛИО;
- сервер может читать Woo order, order items, meta, customer mapping, warehouse mapping;
- сервер может делать idempotency через `externalRequestId`;
- сервер может логировать ошибки и повторять отправку;
- пользователь не сможет случайно подменить критичные поля счета.

При этом endpoint ФОЛИО не должен "сам угадывать" все из Woo. Woo должен отправлять явный, проверенный payload.

## Preview JSON перед отправкой

Первый технический этап должен показать preview JSON без отправки в ФОЛИО.

Preview должен:

- собираться из реального Woo order/draft;
- показывать все поля, которые планируется отправить;
- показывать, какие обязательные привязки отсутствуют;
- показывать распределение по складам;
- показывать `accountingEnabled`;
- показывать `externalRequestId`;
- показывать будущие meta-связи;
- не менять ФОЛИО.

До согласования preview JSON отправку включать нельзя.

## Логирование

Нужен отдельный лог событий интеграции Woo -> ФОЛИО.

Минимальные события:

- preview built;
- missing customer mapping;
- missing warehouse mapping;
- send started;
- send success;
- send failed;
- duplicate/idempotent retry;
- Folio document linked;
- partial order detected.

Лог должен хранить:

- Woo order ID;
- Folio document ID/number, если есть;
- source flow: `cart_to_draft`, `import_to_draft`, `checkout_order`;
- status;
- error code/message;
- timestamp;
- краткий context JSON.
