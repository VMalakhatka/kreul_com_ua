## Практическая заметка: клиенты ФОЛИО для WooCommerce

Для интеграции счетов WooCommerce с ФОЛИО нужен отдельный endpoint списка клиентов/организаций ФОЛИО. Он должен читать `_PARTNER` и, при необходимости, дополнять данные из `_PARTNER_PL`, `VID_DEAT`, `TIP_ORG`, `_PARTNER_TYPES`, `GOROD`.

Первичный фильтр для UI выбора клиента в профиле пользователя Woo:

| `_PARTNER.MY_ORGANIZ` | Значение |
|---|---|
| `Я` | Ваша организация / мои организации |
| `П` | партнер |
| `Д` | дилер |
| `К` | покупатель |
| `Т` | поставщик |
| `I` | иностранный поставщик |

Для выбора клиента Woo в первую очередь показывать партнеров, покупателей и дилеров. Значения `Я`, `П`, `Д`, `К`, `Т` являются кириллическими символами, поэтому нельзя заменять их похожими латинскими буквами. Одна организация может иметь только один такой признак, хотя по документам может фактически выступать в разных ролях.

# 2. Склад (SCL)

Операционный контур ФОЛИО-WinСклад: документы, товары, остатки и служебные механизмы.

## Документы

### Коды типов складских документов

По документации ФОЛИО поле строки `SCL_MOVE.TYPDOCM_PR` должно совпадать с типом документа в заголовке `SCL_NAKL.TYPE_DOC` для того же `UNICUM_NUM`:

```sql
SCL_MOVE.TYPDOCM_PR = SCL_NAKL.TYPE_DOC
```

Возможные значения из документации:

| Код | Unicode | Значение |
|---|---|---|
| `Р` | `U+0420` | расходная накладная |
| `П` | `U+041F` | приходная накладная |
| `С` | `U+0421` | счёт |

Важно: это кириллические буквы `Р`, `П`, `С`, а не латинские `P` и `C`. В интерфейсе они выглядят почти одинаково, но ФОЛИО различает их как разные символы. Ошибка с латинской `C` приводила к созданию SQL-связного счёта, который не отображался как обычный счёт в интерфейсе ФОЛИО.

Для накладной сборки документация описывает особый случай:

- `TYPDOCM_PR = 'П'`, если количество в накладной сборки положительное;
- `TYPDOCM_PR = 'Р'`, если количество в накладной сборки отрицательное.

### SCL_ADDN

- **Статус:** 🟢 Подтверждено экспериментами
- **Назначение:** Дополнительные реквизиты складских документов
- **Описание:** Содержит реквизиты, нужные прежде всего для счетов и счетов-фактур; запись связывается с документом через UNICUM_NUM.
- **Поля (выборочно):** L_ADRESS_P, L_BANK_POR, L_TOWN_POR, L_COD_POR, L_DEB_POR, POLCH_POR, C_ADRESS_P, C_BANK_POR
- **Используется:** Счета, счета-фактуры, адрес доставки, реквизиты плательщика и привязка договора.
- **Участвует в бизнес-процессах:** Счёт, счёт-фактура, создание складского документа через SQL.
- **Связи:** SCL_NAKL по UNICUM_NUM.
- **Подтверждено:**
  - Связана с `SCL_NAKL` по `UNICUM_NUM`.
  - Создаётся одновременно с документом.
- **SQL-проверки:**
  ```sql
  SELECT *
  FROM SCL_ADDN
  WHERE UNICUM_NUM = ...;
  ```

### SCL_MOVE

- **Статус:** 🟢 Подтверждено экспериментами
- **Назначение:** Содержание складских документов
- **Описание:** Содержит товарные строки документов: артикул, требуемое и отпущенное количество, цены, суммы и налоги. Это основной источник движения товаров.
- **Поля (выборочно):** RECNO, UNICUM_NUM, NUM_PREDMT, NAME_PREDM, KOLTREB_PR, KOLC_PREDM, CENA_PREDM, SUM_PREDM
- **Используется:** Проведение документов, расчёт остатков, налогов, цен и складских отчётов.
- **Участвует в бизнес-процессах:** Счёт, приход, расход, перемещение, возврат, списание, инвентаризация, резерв.
- **Связи:** SCL_NAKL по UNICUM_NUM; товар — SCL_ARTC по артикулу и складу.
- **Подтверждено:**
  - Таблица содержит строки складского документа.
  - Каждая строка имеет собственный `RECNO`.
  - `RECNO` является уникальным идентификатором строки.
  - В текущей базе `Paint_Ua` поле `RECNO` является `IDENTITY`: при вставке строки нельзя передавать явное значение `RECNO`, если `IDENTITY_INSERT` выключен. Новый `RECNO` нужно читать сразу после `INSERT` на том же JDBC-соединении.
  - Документ определяется полем `UNICUM_NUM`.
  - Связь с `SCL_ARTC` осуществляется по `(NAME_PREDM, ID_SCLAD)`.
  - Для тестового счёта, прочитанного через Swagger GET, строки содержали `RECNO`, `NUM_PREDMT`, `NAME_PREDM`, `ID_SCLAD`, `KOLC_PREDM`, `CENA_PREDM`, `SUM_PREDM`.
  - `TYPDOCM_PR` имеет тип `varchar(1) NOT NULL`; для счёта подтверждено значение `С` — кириллическая буква Es `U+0421`, не латинская `C`.
  - По документации `SCL_MOVE.TYPDOCM_PR = SCL_NAKL.TYPE_DOC` для строк с тем же `UNICUM_NUM`, кроме особых случаев.
  - `VID_DOC` имеет тип `varchar(20) NULL`; для существующего счёта `UNICUM_NUM = 753524` подтверждено значение `*РАЗОВАЯ`.
  - По `Структура7.doc` часть полей строки переносится из шапки документа: `ORG_PREDM = SCL_NAKL.BRIEFORG`, `NUMDOCM_PR = SCL_NAKL.N_PLAT_POR`, `NUMDCM_DOP = SCL_NAKL.DOPN_SCHET`, `NOT_NAL = SCL_NAKL.NOT_NAL`, `CONTRACT_N = SCL_NAKL.CONTR_POR`, `VALUTROUBL = SCL_NAKL.VALUTROUBL`.
  - `BALL1..BALL5` в `SCL_MOVE` — это суммы баллов строки. Они рассчитываются из `SCL_ARTC.BALL1..BALL5 * KOLC_PREDM`, а не передаются из JavaScript.
  - `VALUT_CENA`, `SUM_VALUT`, `SUM_ROZN` являются строковыми суммовыми полями. Для API они могут передаваться на уровне item; если не переданы, backend пишет безопасные значения `0` для валюты и `price * quantity` для розничной суммы.
- **SQL-проверки:**
  ```sql
  SELECT *
  FROM SCL_MOVE
  WHERE UNICUM_NUM = ...;

  SELECT CAST(@@IDENTITY AS numeric(18,0));
  ```
- **Практические замечания:** При создании документа через SQL строку в SCL_MOVE нельзя рассматривать отдельно от SCL_NAKL и пересчёта остатков/резервов в SCL_ARTC. В текущей базе не вставлять `RECNO` вручную: SQL Server генерирует его сам. Для jTDS/SQL Server 2000 не читать `SCOPE_IDENTITY()` отдельным `JdbcTemplate.queryForObject` после `jdbc.update`: такой вызов может вернуть `NULL`; использовать generated keys или `@@IDENTITY` на том же соединении.

## Товары

### SCL_ARTC

- **Статус:** 🟢 Подтверждено экспериментами
- **Назначение:** Справочник товаров и складских остатков
- **Описание:** Хранит складскую карточку товара, характеристики, цены и текущие количества для конкретного склада. По экспериментам таблица участвует не только как справочник, но и как оперативное место хранения фактического и свободного остатка.
- **Поля (выборочно):** COD_ARTIC, NGROUP_TVR, NAME_ARTIC, CENA_ARTIC, KON_KOLCH, REZ_KOLCH, PRIZN_VALT, CENA_VALT, COD_VALT, NDS_ARTIC
- **Используется:** Карточка товара, документы, остатки, резервы, отчёты, импорт и расширенный каталог SCM.
- **Участвует в бизнес-процессах:** Счёт, резерв, приход, расход, списание, инвентаризация, отчёты по остаткам.
- **Связи:** ALL_ARTC по COD_ARTIC; SCLAD_R по ID_SCLAD; SCL_MOVE по NAME_PREDM и ID_SCLAD; также SCL_SROK, SCL_CODE, SCL_SERT, SCL_PRIC.
- **Подтверждено:**
  - `KON_KOLCH` — фактический остаток.
  - `REZ_KOLCH` — свободный остаток.
  - При создании счёта уменьшается только `REZ_KOLCH`.
  - `KON_KOLCH` при создании счёта не изменяется.
  - Изменение количества в строке документа корректирует `REZ_KOLCH` только на разницу.
  - Удаление строки полностью освобождает резерв.
  - Резерв суммируется по всем активным документам.
  - При попытке создать счёт с товаром `CR-CE0900056045` на складе `7` и `REZ_KOLCH = 0.0` API корректно вернул конфликт недостаточного свободного остатка.
  - `BALL1..BALL5` в карточке товара используются как базовые значения для расчёта сумм `SCL_MOVE.BALL1..BALL5`.
- **SQL-проверки:**
  ```sql
  SELECT KON_KOLCH,
         REZ_KOLCH
  FROM SCL_ARTC
  WHERE COD_ARTIC = '...';

  UPDATE dbo.SCL_ARTC
  SET REZ_KOLCH = REZ_KOLCH - ?
  WHERE COD_ARTIC = ?
    AND ID_SCLAD = ?
    AND REZ_KOLCH >= ?;
  ```
- **Практические замечания:** Это критичная таблица для ручных SQL-операций: документальная строка в SCL_MOVE должна быть согласована с изменением REZ_KOLCH, а KON_KOLCH нельзя менять при обычном создании счёта. При резервировании использовать условный `UPDATE ... AND REZ_KOLCH >= ?` и проверять число обновлённых строк; если обновлено `0`, операция должна откатываться и возвращать ошибку недостаточного остатка.
- **Используется нашим проектом:** контроль свободного остатка; проверка резервов; анализ складских остатков.
- **Источник:** 📖 Документация ФОЛИО — Структура7.doc; FolProf2.DOC; 🔬 подтверждено экспериментами; 🔬 Swagger/API validation test 2026-07-22
